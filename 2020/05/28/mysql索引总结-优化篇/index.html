1. Mysql服务器配置优化(一般是DBA操作，稍作了解)

1.1 通过show status查看mysql服务器运行状态

  • show session status like ‘Com%’; 显示当前的连接的统计结果

  • show global status like ‘Com%’; 显示自数据库上次启动至今的统计结果

  • show status like ‘Connections’; 试图连接MySQL服务器的次数

  • show status like ‘Uptime’; 服务器工作的时间（单位秒）

  • show status like ‘Slow_queries’; 慢查询的次数

    默认是session级别，其中Com_XXX表示XXX语句所执行的次数，比如Com_select，Com_insert，Com_update，Com_delete，可以知道当前数据库是以插入更新为主还是以查询为主，以及各类操作的大致执行比例

1.2 通过show variables查看mysql系统变量

  • show variables like ‘long_query_time’; 慢查询耗时阈值，新版本mysql默认都是1s

1.3 快速定位慢SQL

  • mysql支持把慢查询语句记录到日志文件中，但默认不记录，需自行开启
  • 开启记录慢查询(一般DBA操作)
    • 打开 my.ini，找到 [mysqld]
    • 添加：long_query_time = 2和log-slow-queries = D:/mysql/logs/slow.log，如果不指定记录文件，系统会给一个缺省的记录文件

2.1.2 输出信息字段说明

  1. id：select查询的序列号。表示查询中执行select子句或操作表的顺序，通常有如下三种情况

    1. 如果id都相同，则从上往下执行
    2. 如果id都不相同，则id大的先执行
    3. 如果既有相同又有不同，则id相同的可以看成一组，不同组之间id大的组先执行，同一组之间从上往下执行

  2. select_type：查询的类型。主要用于区别普通查询、联合查询、子查询等复杂查询，常见取值有如下几种

    1. SIMPLE：最简单的查询，不包含UNION查询或者子查询
    2. PRIMARY：查询若包含任何复杂的子部分，最外层查询则被标记为PRIMARY
    3. SUBQUERY：在SELECT或者WHERE列表出现的子查询
    4. DERIVED：在FROM列表出现的子查询，MYSQL会递归执行这些子查询，把结果放进临时表
    5. UNION：若第二个SELECT出现在UNION之后，则被标记为UNION；若UNION包含在FROM子句的子查询中，外层SELECT将被标记为DERIVED
    6. UNION RESULT：从UNION表获取结果的SELECT

  3. table：查询涉及的表

  4. partitions：查询涉及的分区

  5. type：获取数据的方式类型，可判断是否走了索引以及走的索引类型，常见取值如下(性能从上往下逐渐变差)

    1. system：通常出现在系统表，表中只有一行数据的情况，平时见不到
    2. const：通常出现在primary key或者unique key的等值匹配查询场景，表示通过索引一次就可以找到，并且找到数据后就结束扫描直接返回结果。因为只需要匹配一行数据，所以效率极高
    3. eq_ref：通常出现在多表联接查询场景，对于前表的每一条结果数据，都只会匹配后表一条数据，联接的字段有唯一索引或主键索引，并且查询的比较操作通常是=
    4. ref：通常出现在多表联接查询或满足最左前缀的联合索引场景，ref联接的字段一般是普通索引，对于前表每一条记录，都可能在后表存在多条与之匹配的记录
    5. ref_or_null：在ref的基础上还会检索指定字段为NULL的记录
    6. index_merge：通常出现在使用了多个索引的查询，查询结果是从多个索引结果合并得来
    7. unique_subquery：通常出现在IN语句并且IN包含子查询并且IN子查询的结果是主键或唯一键的情况
    8. index_subquery：和unique_subquery一样，但是IN子查询的结果是普通索引
    9. range：通常出现在between、<、<=、>、>=、in范围查询场景，只需要检索指定范围的数据，不用扫描整个索引。当type为range类型时，explain输出的ref字段为NULL，并且key_len字段是此次查询中最长的索引长度
    10. index：通常出现在覆盖索引查询场景，FULL INDEX SCAN，只需要扫描索引就能获得我们查询的字段，此时，explain输出的Extra字段一般是Using index
    11. all：FULL TABLE SCAN，需要遍历整张表，中间可能会有大量磁盘IO，性能最差

  6. possible_keys：查询时可能用到的索引，但不一定被使用

  7. key：查询时真正用到的索引，如果为NULL，则表示没有用到索引(包括没有创建索引或者索引失效)，如果查询时使用了覆盖索引，则该索引仅出现在key列表，不会出现在possible_key列表

  8. key_len：查询使用的索引字节数。可通过该列计算查询中使用的索引长度，在不损失精度的前提下，key_len长度越短越好。key_len显示的值为索引字段的最大可能长度，并非实际使用长度，即key_len是根据表定义计算得出，而不是通过检索出的数据得出。这个字段也可以用来评估联合索引是否被完全使用或者评估索引是否过大。常见数据类型计算规则如下，单位都是字节（注意如果列允许为NULL，则长度额外加1，如果列为可变长度，比如varchar，则长度额外加2）：

    字符串通用规则：(utf8=3,utf8mb4=4,gbk=2,latin1=1) * 列长度 + 1(允许为NULL才加一) + 2(可变长列)

    非字符串通用规则：数值类型或者时间类型 + 1(允许为NULL)

    • 字符串
      • char(n)：n字节长度
      • varchar(n)：如果是utf8编码，则是3 * n + 2字节，如果是utf8mb4编码，则是4 * n + 2
    • 数值类型
      • TINYINT：1字节
      • SMALLINT：2字节
      • MEDIUMINT：3字节
      • INT：4字节
      • BIGINT：8字节
    • 时间类型
      • DATE：3字节
      • TIMESTAMP：4字节
      • DATETIME：8字节

  9. ref：显示与当前索引列匹配的列或者常数const。或者说哪些列或者常量被用于查找索引列上的值

  10. rows：根据表统计信息和索引选用情况，大致估算出找到结果集所需要扫描读取的数据行数，原则上越少越好

  11. filtered：表示符合查询条件的数据百分比，最大100。一般可以和rows一起估算符合条件的记录条数，比如rows = 100，filtered = 50，则估算符合条件的记录数就是100 * 50% = 50

  12. Extra：包含一些额外的重要信息，常见的信息如下

    1. Using filesort(性能极差，极需优化)

      通常出现在使用order by且排序的字段不满足索引条件的查询语句。当查询语句中包含order by并且无法利用索引完成排序的情况下，Mysql Query Optimizer会选择相应的排序算法来实现：数据较少时在内存完成排序，否则从磁盘排序。explain不会显示告诉客户端使用哪种排序，但只要是无法通过索引完成的排序都称为filesort

    2. Using temporary(效率不高，建议优化)

      通常出现在使用order by/group by/多表join的场景，有使用临时表保存结果

    3. Using index

      通常出现在覆盖索引扫描查询。所需结果可以直接从索引树上获取，避免了再次回表读取，效率不错

    4. Using where

      表示使用了where过滤

    5. Using join buffer

      强调在获取联结条件时没有使用索引，并且需要连接缓冲区来缓存中间结果。如果出现这个提示，需要考虑对连接条件添加索引来改进，同时如果查询时多表join次数非常多，需要考虑将配置文件中join buffer缓冲区调大

  13. 注意：要尽量避免让type显示all或者extra提示 using filesort
